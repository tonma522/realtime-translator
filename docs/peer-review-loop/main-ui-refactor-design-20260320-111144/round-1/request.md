```markdown
# Realtime Translator メイン UI リファクタ設計

作成日: 2026-03-20

## 概要

`realtime_translator` の現行メイン UI は、翻訳実行前に確認すべき操作と、低頻度の詳細設定、補助ツールが同じ密度で並んでいる。結果として、開始前判断の導線が散り、翻訳実行中も視線が主結果に集中しにくい。

本設計では、メイン画面を以下の 3 役割へ再編する。

- 左: セッション操作と最小限の実行前判断
- 中央: 翻訳タイムライン
- 右: 補助ツール用の常設ワークスペース

一方で、デバイス選択や詳細な録音・バックエンド設定は既存の設定ウィンドウへ寄せ、主画面の責務を絞る。

## 目的

- 翻訳開始前に必要な判断を主画面の一箇所で完結させる
- 実行中は翻訳結果の視認性を最優先にする
- 補助ツールを維持しつつ、主操作のノイズにしない
- `app.py` に集中している UI 責務を、機能ごとのパネルへ分割する

## 非目的

- バックエンド仕様や翻訳ロジックそのものの変更
- 設定項目の大幅追加
- 新しい補助ツールの追加
- 既存設定ウィンドウの全面 redesign

## 現状課題

- 主画面上部に `聴く / 話す / PTT / 2フェーズ / 原文表示` が横並びし、開始前に読むべき情報と切替操作が混在している
- デバイス選択やモード詳細が設定ウィンドウ側にあり、開始前判断が主画面と分断されている
- `翻訳開始` が主 CTA であるにもかかわらず、他ボタン群と同じ視覚的重みになっている
- 補助ツールが 3 分割で常時表示され、待機中でも画面占有が大きい
- 結果表示、状態表示、補助処理状態の優先順位が UI 上で整理されていない

## 採用レイアウト

比較検討した 3 案のうち、`Focused Console` 案を採用する。

### 採用理由

- このアプリはウィザード型より、常用の翻訳コンソールとしての使い方が中心
- 二方向翻訳の状態を常に見つつ、翻訳結果を広く取る必要がある
- 補助ツールは利用頻度があるため退避せず、ただし 3 分割の同時表示は避けたい

### 不採用案

#### Setup-first Wizard

- 初回導入には強いが、常用時の画面遷移が増える
- 今回の主目的である主画面密度の適正化より、セットアップ誘導に寄りすぎる

#### Dual Stream Studio

- PC音声とマイクの可視性は高い
- ノート PC 幅では横方向の圧迫が強く、中央の読解領域が痩せやすい

## 情報設計

### 左カラム: Main Controls

左カラムには、毎回触る可能性が高い項目のみ残す。

- `翻訳開始 / 翻訳停止`
- `聴く (PC音声)` の有効化
- `話す (マイク)` の有効化
- 翻訳方向サマリー
  - 例: `PC音声: 英語→日本語`
  - 例: `マイク: 日本語→英語`
- 現在モード要約
  - 例: `録音モード: PTT`
  - 例: `翻訳方式: 2フェーズ`
  - 例: `原文表示: ON`
- `詳細設定` ボタン

以下は主画面から外し、設定ウィンドウに残す。

- デバイス選択
- API キー入力
- バックエンド選択
- `PTT / VAD / 2フェーズ / 原文表示` の切替操作そのもの
- 翻訳方向の編集操作そのもの
- Whisper や閾値関連の詳細設定

主画面では詳細値を直接編集させず、要約のみ見せる。これにより「今の設定で開始してよいか」は確認できるが、主導線に低頻度設定を持ち込まない。
翻訳方向の編集場所も `settings_window` に統一し、主画面では方向サマリーのみ表示する。

### 中央: Translation Timeline

中央領域は翻訳結果専用とする。

- 最上部にコンパクトな状態バー
- その下に翻訳タイムライン
- タイムライン各ブロックは以下を持つ
  - ストリーム種別
  - 時刻
  - 原文
  - 訳文
  - エラー時はエラー表示

`show_original=false` の場合は、原文行を省略し、訳文の可読性を優先する。

partial 表示と確定表示の責務は現行と同様に保つが、主画面上での情報優先度は訳文中心に寄せる。

### 右: Workspace Panel

補助ツールは 3 分割同時表示をやめ、常設の単一ワークスペースをタブ切替にする。

タブ構成:

- `再翻訳`
- `返答アシスト`
- `議事録`

採用理由:

- 常時表示要求は満たす
- 3 分割に比べて縦方向の情報量を確保できる
- 処理中・結果表示・エラー表示をタブ内に閉じ込めやすい

## 状態表示設計

### グローバル状態バー

状態表示は中央上部の 1 本へ集約する。優先順位は以下。

1. `エラー`
2. `PTT録音中`
3. `翻訳中`
4. `初期化中`
5. `待機中`

補助ツールの状態は、グローバル状態バーに昇格させない。補助ツール内だけで完結させる。
ここでいう `エラー` は、初期化失敗、翻訳開始失敗、翻訳セッション中の致命的エラーのように、セッション全体の継続可否に影響するものだけを対象とする。再翻訳や議事録生成などタブ内の局所エラーは対象外とし、右タブ内表示で閉じる。

### 左カラム要約との役割分担

- 状態バー: 今この瞬間の実行状態
- 左カラム要約: 現在の構成と開始前判断

これにより、`設定状態` と `稼働状態` を UI 上で明確に分離する。

## 操作フロー

### 待機中

- ユーザーは左カラムで `聴く / 話す` と方向サマリーを確認する
- `翻訳開始` が主 CTA
- 開始不能条件がある場合、開始ボタン近傍に短い不足メッセージを出す
  - 例: `APIキーが未設定`
  - 例: `有効な入力デバイスが未選択`
- 修正先は `詳細設定` に寄せる

### 翻訳中

- `翻訳開始` は `翻訳停止` に反転
- 状態バーは `翻訳中` または `PTT録音中` を表示
- 中央タイムラインが主役
- 右タブは利用可能だが、開始/停止導線を邪魔しない

### 補助作業中

- 右タブで `再翻訳 / 返答アシスト / 議事録` を切り替える
- 補助処理の進行状況はタブ内ラベルやタブ内結果領域で完結させる
- 補助処理が走っても、グローバル状態バーは翻訳実行状態を優先する

## エラーハンドリング

エラーは 2 種類に分ける。

### ブロッカー

翻訳開始できないエラー。左カラムの CTA 近傍に表示する。

対象例:

- API キー未設定
- デバイス未選択
- 必須依存未導入

### 実行時エラー

翻訳中または補助作業中に発生したエラー。中央タイムラインまたは右タブ内に表示する。

対象例:

- 単発の翻訳失敗
- 再翻訳失敗
- 議事録生成失敗

方針:

- `設定画面へ誘導すべきエラー` と `その場で確認できるエラー` を UI 位置で分ける
- 補助ツール起因のエラーは主状態バーを奪わない

## UI コンポーネント境界

レイアウト責務を `app.py` から分離するため、以下の単位で分割する。

### `main_controls_panel`

責務:

- 左カラムの構築
- 開始/停止 CTA
- 有効ストリーム切替
- 方向サマリー表示
- モード要約表示
- 設定画面オープン導線
- 開始前ブロッカー表示

### `translation_timeline_panel`

責務:

- 中央タイムラインの構築
- partial 表示
- 確定翻訳表示
- 原文表示 ON/OFF の反映
- タイムライン上のエラー表示
- 状態バー表示

### `workspace_panel`

責務:

- 右領域の Notebook 管理
- 再翻訳タブ
- 返答アシストタブ
- 議事録タブ
- タブ内ローカル状態表示

現行の `ToolsPanel` は 3 カラム前提のため、Notebook ベースのワークスペースへ再設計する。

### `settings_window`

責務:

- 既存設定編集の継続
- 主画面に残さない全詳細設定の受け皿
- 設定値変更時に、主画面側要約へ反映される情報を提供する

## データフロー

### 設定値

設定ソースは現行どおり `TranslatorApp` 配下の `tk.*Var` を基点とするが、各パネルはそれを直接散発的に読むのではなく、必要情報をメソッドまたは専用更新 API で受け取る。

主な更新方向:

- 設定ウィンドウ -> `TranslatorApp` の変数更新
- `TranslatorApp` -> 左カラム要約更新
- `TranslatorController` -> 中央タイムライン / 右タブ更新

### 実行イベント

キュー処理のハブは `TranslatorApp._poll_queue()` を維持する。ただし、画面反映の責務は各パネルへ委譲する。

例:

- `partial_start` -> `translation_timeline_panel`
- `partial` -> `translation_timeline_panel`
- `translation_done` -> `translation_timeline_panel` と `workspace_panel`
- `retrans_result` -> `workspace_panel`
- `assist_result` -> `workspace_panel`
- `status` -> 状態バー更新

## 実装方針

### 1. レイアウトの枠組みを差し替える

- 現行の上部トグル + 縦 PanedWindow 構成を解体する
- 左・中央・右の 3 領域構成へ置き換える

### 2. `ToolsPanel` を Notebook 化する

- 再翻訳、返答アシスト、議事録をタブにする
- 各タブの既存ロジックはできるだけ流用する

### 3. 状態表示を 1 本化する

- 現在の `_status_var` を中央状態バーの主表示にする
- 補助ツールの局所状態は各タブ内に閉じ込める

### 4. 左カラム要約を追加する

- `PTT / VAD / 2フェーズ / 原文表示 / 翻訳方向` は直接操作ではなくサマリー表示にする
- 設定変更後に即反映されるようにする

### 5. バリデーション導線を整理する

- `_start_inner()` の事前条件エラーは、可能な限り開始ボタン近傍メッセージとして出す
- 既存の例外処理やログ処理は維持する

- タブ内エラーが他タブや主状態バーを壊さないこと
```
