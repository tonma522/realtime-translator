## レビュー結果

---

### Findings（重要度順）

---

#### [中] 1. パネル更新 API のメソッドシグネチャが未定義

**仕様の抜け（前回 [高]2 の部分解決残）**

メソッド名の列挙はされたが、各引数の型・内容が書かれていない。例：

- `apply_session_summary(...)` — 何を渡すか不明
- `on_translation_done(...)` — 既存 `translation_done` イベントペイロードと対応するか不明
- `set_blocker(message | None)` — `message` が文字列か構造体かが不明

Step 1 で「API 経由で既存 UI を更新できること」を検証ゲートとしている以上、引数仕様のない API はその時点で実装者が独自判断に陥る。

---

#### [中] 2. 既存 `error` イベントの正規化ロジックが未定義

**バグ・回帰リスク**

「既存の単純な `error` イベントを残す場合でも `TranslatorApp` 内で上記形式へ正規化してから渡す」とあるが、正規化のマッピング規則が書かれていない。既存イベントの `scope / severity / source` 値を何で決定するか（イベント名？payload の既存フィールド？）が実装者任せになっており、誤分類リスクがある。

---

#### [中] 3. Step 1 検証ゲートとステップ内容が矛盾

**テスト不足**

Step 1 の内容は「パネル間更新 API と状態リゾルバを先に導入する」であり、各パネルクラスはまだ存在しない。しかし検証ゲートは「新 API 経由で既存 UI が更新できること」となっている。パネルが存在しない段階で何に向けて API を呼ぶのか（スタブ？既存 widget を直接ラップ？）が不明で、Step 1 の検証が実施不能になるリスクがある。

---

#### [低] 4. 未解決事項 3 点の確定を阻止するゲートがない

**完了条件の不備**

完了条件に「未解決事項 3 点が実装着手前に確定されること」が追加されたが、段階的実装のどのステップより前に確定するかが書かれていない。Step 1 が始まる前の前提条件として明示されなければ、実質的なゲートとして機能しない。

---

#### [低] 5. 完了条件「退行していない」の定性性が残存

**完了条件の不備（前回 [低]9 の部分改善残）**

各ステップの検証ゲートで翻訳・再翻訳・議事録の動作確認が追加されたが、最終完了条件の文言「既存の翻訳、再翻訳、返答アシスト、議事録機能が退行していない」はステップのゲートリストへの明示的な参照がない。どのゲート項目が最終判定に対応するかが明確でなく、検収時に属人的判断が残る。

---

### 重大指摘

なし（前回の [高] 指摘 3 点はすべて実質的に解消された）

---

### 総評

前回の [高] 指摘3点（エラー分類契約・パネル API・検証ゲート）はいずれも具体的な記述で対応されており、構造的な改善は十分。残課題はメソッドシグネチャの未記述と Step 1 検証ゲートの論理的矛盾に絞られており、実装着手前にこの2点を補足すれば仕様として実用可能な水準に達している。
