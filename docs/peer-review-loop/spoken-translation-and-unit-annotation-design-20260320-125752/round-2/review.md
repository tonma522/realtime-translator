## Findings（重要度順）

---

### [高] 返答アシスト・議事録の除外ロジックが未定義

`Integration Point` では「返答アシストと議事録生成には本後処理を適用しない」と明記されているが、`_poll_queue()` の `translation_done` 分岐でどのように除外するか（判定フラグ、メッセージ種別の識別方法など）が一切書かれていない。  
もし両者も `translation_done` を発火するなら、除外なしで `annotate_translation()` が適用されてしまう。除外の具体的な手段を仕様に記載し、Done Criteria と integration test に「返答アシスト・議事録経路では後処理が適用されないこと」を追加する必要がある。

---

### [高] `annotate_translation()` の `source_stream_id` / `virtual_stream_id` 引数の目的が未定義

API シグネチャに `source_stream_id: str` と `virtual_stream_id: str | None` が含まれるが、後処理がこれらをどう使うかが一切説明されていない。  
後処理が pure function であるという設計方針（LLM 非決定性に依存しない）と矛盾しうる。v1 で使わないなら引数を持つべきでなく、将来用途なら別途 `context` オブジェクトへの包含を検討すべき。現状は「意図不明な引数が公開 API に露出している」状態で、実装者が誤ったロジックを混入するリスクがある。

---

### [高] 「同一テキストへ二重適用しない」の保証方法が未定義

仕様として要件は明記されているが、どうやって二重適用を防ぐかが書かれていない。  
`annotate_translation()` の冪等性（すでに注釈済みのテキストを誤って二重注釈しない）を関数レベルで保証するのか、呼び出し側の契約で保証するのかが不明。unit test の観点にも含まれていない。

---

### [中] 圧力の複数換算先（`MPa / bar` 併記）の選択ロジックが未定義

`35 psi (0.24 MPa / 2.41 bar)` という例が示されているが、なぜ圧力だけ複数単位を並記するのか、他の単位（例: `Nm`）には適用しないのか、その条件と優先順位が仕様化されていない。実装者によって動作が変わるリスクがある。

---

### [中] `AnnotationResult` の型定義がない

`convert_engineering_token` と `convert_abrasive_token` の返り値が `AnnotationResult | None` とされているが、その構造（フィールド名・型）が仕様に記載されていない。API 設計の共通理解がないまま実装が分岐するリスクがある。

---

### [中] 裸数字の「話される数値」判定基準が実装依存のまま

`Number Reading Policy` に「文脈上の数量表現と判定できる場合にのみ読みを付ける」とあるが、その判定を純粋な正規表現・位置情報で行うのか、形態素解析的なアプローチが必要なのかが不明。後処理が pure function を前提とするなら、LLM への問い合わせなしにどこまで判定できるかを仕様上で制約する必要がある。

---

### [低] テスト観点に範囲・公差表現の換算確認がない

仕様本文では「範囲表現は両端を換算する」「公差表現は符号を保持して換算する」と明記されているが、Test Strategy のテスト観点リストにこれらが含まれていない。Done Criteria も同様に不足している。

---

### [低] `build_number_reading` を public API として公開する必要性が不明

この関数は `convert_*_token` の内部実装から呼ばれるだけであれば、外部公開する理由がない。テストのために公開するとしても、その意図を仕様に記載すべき。過剰な public surface になる。

---

## 重大指摘なし → 該当せず

高重要度の指摘が3件残っているため「重大指摘なし」とは書けません。

---

## 総評

前回指摘の主要3点はすべて解消されており、仕様の完成度は大幅に向上している。残る重大リスクは「返答アシスト・議事録の除外ロジック未定義」「stream ID 引数の意図不明」「二重適用防止の保証欠如」の3点で、いずれも Integration Point 周辺の記述を補完することで解消できる。`AnnotationResult` 型定義と圧力複数換算先の条件整理を合わせて対処すれば、実装着手に十分な水準に達する。
