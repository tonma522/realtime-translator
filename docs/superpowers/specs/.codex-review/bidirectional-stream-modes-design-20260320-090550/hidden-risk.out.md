[High] `auto` の `DIRECTION:` 先頭行固定を前提にしているが、モデル/プロバイダ別の実ストリームで契約確認するリリースゲートがなく、一部バックエンドだけ恒常的に `direction_parse_failed` 化するリスク。  
[High] 仮想ストリーム化を controller・UI・履歴・prompt に同時投入する計画だが、機能フラグや段階展開がなく、既存 `listen` / `speak` 回帰時の切り分けと安全な rollout が難しい。  
[High] 新形式の設定値・履歴を保存した後に旧版へ戻す手順が未定義で、ロールバック時に旧バイナリが `pc_audio_mode` / `mic_mode` や仮想 `stream_id` をどう扱うかが不明。  
[Medium] 旧履歴はメモリ補完のみで永続バックフィルしないため、セッションを跨ぐたびに同じフォールバックを繰り返し、表示ラベルや再翻訳結果の差異を追跡しにくい。  
[Medium] `direction_parse_failed` を参照専用にする設計だが、手動再試行や後日バックフィル手段がなく、一時的なモデル逸脱で履歴欠損が恒久化するリスク。  
[Medium] STT 言語コードを `en` / `ja` の2値前提で扱っており、`en-US` `ja-JP` `unknown` など実際の返却値をどう正規化するかの前提を明示すべき。  
[Medium] `direction_parse_failed` 件数、STT判定利用率、LLMフォールバック率、方向確定までの遅延などの監視項目がなく、`auto` 品質低下の検知が遅れる。  
[Medium] 設定不正値を既定値へ黙って正規化する設計で、意図しない翻訳方向へ落ちても利用者や運用者が気づきにくいため、警告表示やログ出力の前提を明示すべき。  
[Low] PTT は常に `mic_mode` を共有する前提だが、PTT だけ別方向を期待する運用を非対応とする判断理由と受け入れ条件が明示されていない。
