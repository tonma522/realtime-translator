# 双方向ソース別翻訳モード設計

作成日: 2026-03-20

## 概要

PC音声とマイクの各入力ソースに対して、以下の3モードを独立に設定できるようにする。

- 英語→日本語
- 日本語→英語
- 同時翻訳

ここでの「同時翻訳」は、1回の音声から両言語を同時出力する意味ではない。入力音声が英語か日本語かを自動判定し、反対言語へ1本だけ翻訳する意味で扱う。

本設計では、既存の `listen` / `speak` をそのまま翻訳方向の識別子として使わず、入力ソースごとに仮想ストリームを導入する。

## 目的

- マイク入力でも英語→日本語を扱えるようにする
- PC音声でも日本語→英語を扱えるようにする
- 各入力ソースで固定方向または自動方向判定を選べるようにする
- 既存の音声キャプチャ構造とUI色分けを大きく崩さずに拡張する

## 要件

### 機能要件

- `PC音声` に対して以下を選択可能にする
  - 英語→日本語
  - 日本語→英語
  - 同時翻訳
- `マイク` に対して以下を選択可能にする
  - 英語→日本語
  - 日本語→英語
  - 同時翻訳
- 既存の `聴く` / `話す` の有効化は維持する
- 同時翻訳時は、話された主要言語を英語または日本語のどちらかとして扱い、反対言語へ翻訳する
- 通常モード、2フェーズ、Whisper、外部STTの各経路で方向指定を扱えること

### 非機能要件

- 既存の履歴、再翻訳、返答アシスト、議事録生成の前提を壊さない
- UI変更は既存レイアウトの延長で理解しやすいものにする
- 方向判定ロジックはプロンプト中心で実現し、ローカル側で複雑な言語判定器を持たない
- `auto` モードでも体感速度を優先し、固定方向モードに近い応答感を維持する

## 採用方針

仮想ストリーム方式を採用する。

### 仮想ストリーム定義

PC音声:

- `listen_en_ja`
- `listen_ja_en`
- `listen_auto`

マイク:

- `speak_en_ja`
- `speak_ja_en`
- `speak_auto`

この方式では、入力ソースと翻訳ルールを固定ペアにする。既存のプロンプト生成、履歴表示、再翻訳は最終的に仮想ストリームIDを見て動作するように寄せる。

## 他案と不採用理由

### 案1: ソース別モード値を別途持ち、`listen` / `speak` は維持する

- 利点: 既存構造への変更量が少ない
- 不採用理由: `stream_id` 以外の別情報を複数箇所に渡す必要があり、履歴、プロンプト、表示ラベル、再翻訳の責務境界が曖昧になる

### 案2: 入力言語と出力言語を完全に汎用化する

- 利点: 将来拡張性は高い
- 不採用理由: 今回要件に対して抽象化が強すぎ、UIと設定が過剰になる

## UI設計

### 有効ストリーム

既存の以下は維持する。

- `聴く`
- `話す`
- `プッシュ・トゥ・トーク`
- `2フェーズ(STT→翻訳)`
- `原文も表示`

### 設定ウィンドウの追加項目

デバイス設定または翻訳設定付近に、以下のコンボボックスを追加する。

- `PC音声モード`
  - 英語→日本語
  - 日本語→英語
  - 同時翻訳
- `マイクモード`
  - 英語→日本語
  - 日本語→英語
  - 同時翻訳

既定値:

- PC音声: 英語→日本語
- マイク: 日本語→英語

これにより既存利用者の初期挙動を極力維持する。

Whisper / OpenAI STT / OpenRouter STT のように `2フェーズ` と両立しない STT バックエンドを選んだ場合、UI は既存挙動どおり `2フェーズ` を無効化または強制オフにする。設定ウィンドウ上でも、その制約が分かる表示にする。

## 表示設計

### 実行中表示

結果表示には、ソース名と最終的な方向を出す。

例:

- `PC音声 英語→日本語`
- `PC音声 日本語→英語`
- `マイク 英語→日本語`

`同時翻訳` を選択している場合でも、結果表示や履歴には最終採用方向を出す。`同時翻訳` という設定名はUI設定値として使い、出力上は具体方向を優先する。

方向確定前の partial 表示や処理中表示は、`PC音声 同時翻訳` / `マイク 同時翻訳` のようにモード名を表示し、確定後に `英語→日本語` または `日本語→英語` へ置き換える。

速度優先方針として、`auto` モードでは全文受信完了まで待たない。方向確定と訳文 streaming 開始の条件は `resolved_direction` の確定のみとし、原文取得完了は待たない。通常モードの `auto` では先頭 `DIRECTION` 行のパース完了時点、STT 言語情報がある経路ではそのメタデータ受信時点で方向を確定し、以後の訳文は遅延なく順次表示する。

`show_original=true` の場合でも、原文は訳文とは独立に後着反映してよい。`show_original=false` の場合は原文を UI に流さない。原文取得のために訳文開始を遅らせる設計は採らない。

### 履歴

履歴エントリは新旧互換を維持したまま仮想ストリームIDを保持できるようにする。これにより再翻訳時も同じ方向情報を失わない。

履歴には少なくとも以下を保持する。

- `stream_id`
- `virtual_stream_id`
- `resolved_direction`
- `original`
- `translation`

`stream_id` は互換維持用の入力ソース識別子として今後も保持し、値は常に `listen` または `speak` とする。UI色分け、入力ソース表示、旧呼び出し側互換には `stream_id` を使ってよい。一方、翻訳方向の解決には `stream_id` を使わず、`resolved_direction` を最優先し、その次に `virtual_stream_id` を使う。

`virtual_stream_id` は `listen_en_ja` / `listen_ja_en` / `listen_auto` / `speak_en_ja` / `speak_ja_en` / `speak_auto` のいずれかを取り、`resolved_direction` は最終確定した `en_ja` または `ja_en` を保持する。`auto` で方向確定に失敗した場合のみ `resolved_direction=null` を許容する。

既存の履歴エントリ互換のため、`virtual_stream_id` と `resolved_direction` が存在しない旧形式データは以下へフォールバックする。

- `listen` -> `listen_en_ja`
- `speak` -> `speak_ja_en`

再翻訳や表示ラベル生成では、まず `resolved_direction` を優先し、その次に `virtual_stream_id`、どちらも欠損している旧形式データにのみこのフォールバック規則を使う。

旧形式履歴エントリを再翻訳した際は、解決した方向をメモリ上のエントリへ補完して以後の再利用に使う。永続化済み旧データの一括マイグレーションは行わない。

## データモデル変更

### 設定値

新規に以下を追加する。

- `pc_audio_mode`
- `mic_mode`

値は以下のいずれかとする。

- `en_ja`
- `ja_en`
- `auto`

不正値正規化は `normalize_translation_mode(value, default)` のような単一ヘルパーで行い、保存前と読込後の両方から同じヘルパーを呼ぶ。これにより JSON に不正値を残しにくくしつつ、既存設定読込時の防御も維持する。

実装時は既存の設定書き込みパスを棚卸しし、`pc_audio_mode` / `mic_mode` を保存する経路がすべてこのヘルパーを通ることを確認する。

不正値を既定値へ正規化した場合は、少なくとも debug ログまたは診断ログへ `normalized_invalid_translation_mode` を残す。設定UIが開いている場合は、次回保存時に正規値へ戻されたことが分かる表示を行ってよい。

### メタ定義

既存の固定 `STREAM_LANGS` と `_STREAM_META` は、仮想ストリーム単位の辞書へ拡張する。

保持したい情報:

- 入力ソースラベル
- 表示タグ
- 固定方向か自動方向か
- 固定方向時の `src` / `dst`
- 表示用ラベル

## 制御フロー設計

### 起動時

`StartConfig` に以下を追加する。

- `pc_audio_mode`
- `mic_mode`

`StartConfig` 側でも既定値を持たせる。

- `pc_audio_mode="en_ja"`
- `mic_mode="ja_en"`

コントローラー起動時に、入力ソースごとの設定から実際に使う仮想ストリームIDを決定する。

例:

- PC音声 + `en_ja` -> `listen_en_ja`
- PC音声 + `ja_en` -> `listen_ja_en`
- PC音声 + `auto` -> `listen_auto`
- マイク + `en_ja` -> `speak_en_ja`
- マイク + `ja_en` -> `speak_ja_en`
- マイク + `auto` -> `speak_auto`

### 音声チャンク受付

音声キャプチャ自体は従来どおりソース単位で作成する。コールバックで仮想ストリームIDを解決して、以後はそのIDを `ApiRequest` へ渡す。

PTT はマイク入力専用のため、常に `mic_mode` から解決された `speak_*` 仮想ストリームを使う。PTT 専用の別モードは持たない。

設定UI上でも、PTT は `マイクモード` をそのまま使うことが分かる説明文を添える。PTT 専用の翻訳方向設定は追加しない。

### 通常モード

固定方向:

- 仮想ストリームIDから `src` / `dst` を取得し、既存どおり1回の呼び出しでSTT+翻訳を行う

自動方向:

- 音声内容が英語か日本語かを判定し、反対言語へ翻訳するよう明示した専用プロンプトを使う
- 出力には原文と訳文、または訳文のみを返す
- 結果ラベルには最終採用方向を含める
- LLM 直結経路では、速度優先のため応答は `DIRECTION` 行を必ず先頭に出し、その後ろに `TRANSLATION` を続ける

### 2フェーズ

Phase1:

- 自動方向でも固定方向でも、まず文字起こしを行う
- `auto` モードで Whisper / 外部STT を使う場合、STT の言語パラメータは未指定として自動判定に委ねる。`ja` / `en` の固定ヒントは渡さない

Phase2:

- 固定方向では従来どおり `src` / `dst` を指定して翻訳する
- 自動方向では STT 言語情報が取得できる場合、その値から `resolved_direction` を確定して固定方向翻訳へ入る
- STT 言語情報が取得できない場合のみ、文字起こし結果を見て英語または日本語を判定し、反対言語へ翻訳する専用テキストプロンプトで方向を確定した後に固定方向翻訳へ入る
- Phase2 の共通化対象はテキスト wire format ではなく、`resolved_direction`、`original`、`translation_delta`、`translation_final` の内部イベントとする

### Whisper / 外部STT

- STTワーカーへの投入は仮想ストリームID単位にする
- STT後の翻訳処理で仮想ストリームIDに応じた固定方向または自動方向プロンプトを使う
- 既存挙動を維持し、Whisper または外部STT 使用時は `2フェーズ` を無効化する。つまり `Whisper/OpenAI STT/OpenRouter STT + 2フェーズ` の組み合わせは新たに定義せず、STT結果をそのまま翻訳段へ渡す1経路に統一する
- 速度優先のため、Whisper / OpenAI STT / OpenRouter STT が言語情報を返せる場合、`auto` の方向決定にはその検出言語を優先利用し、LLM に再判定させない
- 外部STTが言語情報を返せない場合のみ、既存の `auto` 用 LLM プロンプト判定へフォールバックする
- 全経路は最終的に `resolved_direction`、`original`、`translation_delta`、`translation_final` を同じ内部インターフェースへ流す。必要に応じて `direction_source` として `llm_header` / `stt_metadata` / `llm_fallback` を内部保持する

STT 言語情報から `resolved_direction` への変換は以下で固定する。

- `en` -> `en_ja`
- `ja` -> `ja_en`
- それ以外または未取得 -> LLM 判定へフォールバック

STT 言語コードは比較前に正規化する。少なくとも `en-US` / `en-GB` は `en`、`ja-JP` は `ja` へ畳み込み、`unknown` / 空文字 / `None` は未取得として扱う。

## プロンプト設計

### 固定方向

対象:

- `*_en_ja`
- `*_ja_en`

既存の `build_prompt` / `build_stt_prompt` / `build_translation_prompt` / `build_retranslation_prompt` の考え方を維持しつつ、仮想ストリームIDから `src` / `dst` を解決する。

固定方向モードの通常出力は既存どおりで、`DIRECTION` 行を含めず、翻訳文または `原文` / `訳文` の既存フォーマットを返す。

### 自動方向

対象:

- `*_auto`

専用プロンプトで以下を明示する。

- 入力が英語か日本語かを判定すること
- 判定した主要言語が英語なら日本語へ翻訳すること
- 判定した主要言語が日本語なら英語へ翻訳すること
- コードスイッチング時は主要言語を採用すること
- 出力は1つだけ返すこと

モデル応答から方向を確実に抽出できるよう、`auto` 用の応答契約を設計段階で固定する。

経路ごとの契約は次で固定する。

1. LLM 直結 `auto`

```text
DIRECTION: en_ja | ja_en
TRANSLATION: <translated text>
```

`DIRECTION:` は応答の先頭行に固定し、最初に生成させる。パーサはこの先頭行を受け取った時点で方向を確定し、残りの `TRANSLATION` テキストは全文完了を待たずに逐次表示する。`ORIGINAL` は streaming 開始条件にしない。

`show_original=true` で原文が必要な場合は、完了後メタデータまたは別フィールドで取得する。`show_original=false` の場合は原文を UI に流さない。

2. STT 確定 `auto`

翻訳 LLM には固定方向プロンプトを渡し、出力は既存の固定方向フォーマットまたは訳文 streaming のみとする。この経路では `DIRECTION` 行を要求しない。

固定方向モードは既存フォーマットを維持してよい。`auto` の共通化対象は wire format ではなく、最終的に `resolved_direction` を得て訳文を流す内部イベントである。

`*_auto` の再翻訳では、履歴に保存された `resolved_direction` を用いて固定方向の再翻訳として扱う。つまり再翻訳時に再度言語判定は行わない。`resolved_direction` が欠損している旧データのみ、旧フォールバック規則または `virtual_stream_id` から方向を解決する。

再翻訳プロンプト設計も同様で、`*_auto` 専用の再翻訳フォーマットは持たない。再翻訳時は `resolved_direction` に応じて `en_ja` または `ja_en` の固定方向 `build_retranslation_prompt` を使う。

streaming 応答では、`DIRECTION` の確定は先頭行の受信完了かつパース成功時点とする。トークン途中や行途中では方向確定しないが、先頭行確定後は `TRANSLATION` を逐次表示してよい。

パーサ仕様は次の通りとする。

1. 受信テキストは先頭改行まで専用バッファへ蓄積する
2. 先頭改行は `\\n` / `\\r\\n` の両方を許容し、先頭行は末尾改行と前後空白を trim した上で `DIRECTION: en_ja` または `DIRECTION: ja_en` と一致した場合のみ方向確定とみなす
3. 方向確定後は `TRANSLATION:` プレフィックスを検出した時点で、その後続テキストを streaming 表示へ流す
4. `ORIGINAL` を扱う実装でも、それは表示開始条件にせず、内部メタデータとしてのみ処理する
5. 先頭行が不正、または `TRANSLATION:` セクションへ遷移できない場合はフォーマット逸脱エラーとする

フォーマット逸脱エラー時は、パース専用バッファを UI へフラッシュしない。`original` を別途取得できている場合のみ `original` フィールドへ採用し、そうでなければ `original=""` とする。

`DIRECTION` 先頭行の待機は既存の API / worker タイムアウトに従う。`auto` 専用の追加タイムアウトは設けない。

## エラーハンドリング

- モード未設定時は既定値にフォールバックする
- 不正なモード文字列は保存時または読込時に既定値へ正規化する
- 自動方向で主要言語を決定できない、または構造化応答を正しく返せない場合はエラーとして扱う
- 既存のデバイス未選択、APIキー未設定、依存未導入のエラー処理は維持する
- 実行中に `pc_audio_mode` / `mic_mode` を変更しても、反映は次回 `翻訳開始` 時とする。稼働中キャプチャへの即時反映は行わず、UIには「次回開始時に反映」の案内を出す

LLM 直結 `auto` の構造化応答で `DIRECTION` が欠損または `en_ja` / `ja_en` 以外だった場合は、推測補完せずエラーとして扱う。ただし音声や文字起こし結果は破棄せず、`stream_id` と `virtual_stream_id` を保持したまま、`resolved_direction=null`、`translation=""`、`error="direction_parse_failed"` を持つ不完全履歴として残し、UI にはフォーマット逸脱エラーを表示する。`original` は取得済みの場合のみ保持する。

不完全履歴エントリは自動再翻訳の対象にしない。再翻訳ボタンは無効化し、将来手動リトライ機能を追加するまでは参照専用とする。

`direction_parse_failed` な不完全履歴は、返答アシストの入力履歴、議事録生成の本文候補、既定のエクスポート本文には混入させない。診断用途の生履歴エクスポートでのみ、`error` 付きレコードとして含めてよい。

`direction_parse_failed` な履歴エントリは、履歴一覧で通常エントリと区別できる見た目にする。少なくともエラーラベル表示と再翻訳無効状態を持たせる。配色やアイコンは既存 UI パターンに合わせる。

## テスト方針

### prompts

- 仮想ストリームIDごとに正しい言語方向が埋め込まれること
- `auto` 用プロンプトに「言語判定して反対言語へ翻訳」が含まれること
- LLM 直結 `auto` 用応答フォーマットに `DIRECTION` / `TRANSLATION` が含まれること
- STT 確定 `auto` では固定方向翻訳プロンプトへ切り替わり、`DIRECTION` ヘッダを要求しないこと
- `*_auto` の再翻訳プロンプトが `resolved_direction` を使う固定方向再翻訳へ切り替わること

### controller

- ソースごとのモード設定から正しい仮想ストリームIDが選ばれること
- 通常モード、2フェーズ、Whisper、外部STTの各経路で正しいIDが流れること
- `auto` で確定した `resolved_direction` が履歴へ保存され、再翻訳時に再利用されること
- `auto` で `DIRECTION` パースに失敗した場合、`direction_parse_failed` な不完全履歴を残しつつエラー通知されること
- `auto` で先頭 `DIRECTION` 行受信後に、`TRANSLATION` が streaming 表示へ移行すること
- 外部STT が言語情報を返す場合、`auto` でその言語情報を優先して `resolved_direction` を決めること
- 外部STT が言語情報を返さない場合、`auto` で LLM 判定へフォールバックすること
- 旧形式履歴エントリの再翻訳時に、フォールバック規則で方向解決され、メモリ上の `resolved_direction` が補完されること
- `direction_parse_failed` な不完全履歴エントリは再翻訳対象から除外されること
- 新規履歴で `stream_id=listen/speak` が維持されつつ、方向解決は `resolved_direction` と `virtual_stream_id` を優先すること
- `direction_parse_failed` な不完全履歴が返答アシスト、議事録生成、既定エクスポート本文から除外されること

- `pc_audio_mode` と `mic_mode` の保存・読込ができること
- 不正値が既定値へ正規化されること
- 旧設定ファイルにキーがない場合、PC音声=`en_ja`、マイク=`ja_en` にフォールバックすること
- 不正値正規化時に診断ログが残ること

### UI

- 設定ウィンドウのモード選択が初期値を反映すること
- 結果表示ラベルが仮想ストリーム定義に従うこと

- 実行中にモード変更した場合、「次回開始時に反映」の案内が表示されること

## リリース・移行方針

### 段階展開

- 仮想ストリーム機能は、実装時に機能フラグで段階展開可能にする
- 初期リリースでは、設定UI、controller、履歴保存、再翻訳導線を同時有効化しない構成も選べるようにする
- `auto` モードはバックエンドごとの契約確認が済んだ組み合わせから順に有効化する

### リリースゲート

- LLM 直結 `auto` を有効化する各プロバイダ / モデル組み合わせについて、実ストリームで `DIRECTION:` 先頭行契約を満たすスモークテストを持つ
- スモークテストでは、少なくとも英語入力1件と日本語入力1件で `direction_parse_failed` にならず、先頭行確定後に `TRANSLATION` が streaming されることを確認する
- 契約未確認のバックエンドでは `auto` を無効化するか、固定方向のみを提供する

### ロールバック互換

- 永続化データでは既存 `stream_id` を `listen` / `speak` のまま維持し、仮想ストリームIDは追加フィールド `virtual_stream_id` として保存する
- 旧版アプリへロールバックした場合でも、未知の設定キー `pc_audio_mode` / `mic_mode` と追加履歴フィールドを無視できることを前提にする
- 旧版で読めない形式へ既存フィールドを上書きしない。特に `stream_id` へ `listen_en_ja` のような新値は保存しない

### 監視

- 少なくとも `direction_parse_failed` 件数、`direction_source` 内訳、STT メタデータ利用率、LLM フォールバック率、`resolved_direction` 確定までの遅延、最初の訳文表示までの遅延を診断ログまたはメトリクスで観測できるようにする
- 速度優先要件の劣化を継続監視できるよう、固定方向 streaming 比の遅延比較を定期計測またはリグレッションテストで保持する
