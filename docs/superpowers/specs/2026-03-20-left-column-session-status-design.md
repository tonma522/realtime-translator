# Left Column Session Status Design

作成日: 2026-03-20

## 概要

メイン画面左カラムの下側に大きな空きが残っており、視覚的な密度が不均衡になっている。
この空きは単なる余白削減ではなく、左カラム本来の役割である「開始前判断」と「セッション構成の確認」を強める形で埋める。

本設計では、左下領域を `セッション構成` と `クイック操作` の 2 ブロックで構成する。
`セッション構成` は開始前判断に必要な固定要約だけを扱い、実行状態やエラー表示は親設計どおり中央状態バーと開始前ブロッカーへ残す。
右側ワークスペースのタブ本体は移さず、左カラムはあくまでセッション構成の確認と低頻度の補助操作に集中させる。

## 目的

- 左下の空きを意味のある情報で埋める
- 左カラムの役割を `開始前判断 + セッション構成の確認 + 軽操作` に統一する
- 右ワークスペースとの責務重複を避ける
- 実行状態の表示責務を中央状態バーと競合させない

## 非目的

- 右タブ機能の縮小コピーを左カラムへ持ち込むこと
- 中央状態バーの代わりに実行状態や実行時エラーを左下へ持ち込むこと
- 詳細設定項目の再配置
- 左カラムを操作ボタンだらけにすること

## 採用方針

左下は次の 2 ブロック構成とする。

1. `セッション構成`
2. `クイック操作`

優先順位は `セッション構成` が主、`クイック操作` が従である。
左下にタブ要約を置く案も検討したが、右ワークスペースと役割が近く、左カラムの主旨がぼやけるため採用しない。
中央状態バーとの責務衝突を避けるため、左下では実行中に変化する状態を扱わない。

## 情報設計

### 上部と下部の役割分担

- 左カラム上部は `開始/停止`、`聴く (PC音声)`、`話す (マイク)`、`詳細設定`、有効ストリーム、モード要約を扱う
- `聴く (PC音声)` と `話す (マイク)` は、主画面で常時切替可能なコア操作として上部に残し、左下には移さない
- 左カラム下部の `セッション構成` は、開始前判断に必要な固定情報だけを扱う
- 親設計で左カラム要件に含めていた `方向サマリー` は、左下の `セッション構成` の一部として表示する
- 実行状態、実行時エラー、開始前ブロッカーは左下へ複製せず、中央状態バーまたは開始ボタン近傍に残す
- 既存の `クリア` / `エクスポート` はヘッダーから左下の `クイック操作` へ寄せる

### セッション構成

表示候補は以下に限定する。

- 方向サマリー
  - 例: `PC音声: 英語→日本語`
  - 例: `マイク: 日本語→英語`
  - 有効ストリームの ON/OFF はここに重ねず、上部の有効ストリーム表示で扱う

- 使用デバイス
  - 例: `PC音声: Speakers (Loopback)`
  - 例: `マイク: Yeti Nano`
- 使用バックエンド
  - 例: `STT: Gemini / 翻訳: Gemini`
- 構成更新時刻
  - 例: `構成更新: 15:42:18`

表示ルール:

- `セッション構成` は開始前判断に必要な固定情報だけを扱い、`状態: 待機中` や `最終エラー` のような実行状態は置かない
- 値が未確定の場合は空欄にせず、`未設定` または `未取得` を表示する
- `構成更新時刻` は、初期設定読み込み、設定値の確定、デバイス再読込成功のいずれかで更新する
- 設定値の確定は、設定ウィンドウ close 時ではなく、`trace` コールバックまたは明示コールバックで `TranslatorApp` 側の値が確定した時点を指す
- 翻訳イベントや補助ツールイベントでは `構成更新時刻` を更新しない

ここでは編集操作を持たせず、表示専用とする。詳細修正が必要な場合は既存の `詳細設定` へ誘導する。

### クイック操作

左下に置くクイック操作は、左カラムから起動しても意図が明確で、`TranslatorApp` のコールバック経由で安全に実行できるものだけに限定する。
他パネルの内容編集やワークスペース切替を直接担う操作は置かない。

配置する操作は多くても 3 個までとし、各操作について `表示ラベル`、`アイコン有無`、`有効条件`、`無効時の見せ方`、`失敗時の表示先` を固定する。

共通表示ルール:

- ボタンラベルはテキストのみとし、アイコンは付けない
- 見た目は主 CTA より弱い標準 `ttk.Button` とし、色で主従を逆転させない
- ボタンが無効な場合は非活性表示に加えて、直下の補助文で理由を短文表示する
- 補助文は常時 1 行だけ確保し、最新の理由または失敗メッセージで上書きする
- 補助文の例は `初期化中`、`翻訳中は使えません`、`結果がありません`、`保存に失敗しました`

- `デバイス再読込`
  - 対象: 入力・出力デバイス一覧
  - 実行: `TranslatorApp._refresh_devices()` を呼び、`セッション構成` も更新する
  - 有効条件: 初期化が完了しており、翻訳が停止中であること
  - 補足: `PTT 録音中` は翻訳実行中に含め、使用不可とする
  - 確認要否: 不要
  - 無効時の見せ方: ボタンを非活性にし、補助文へ理由を表示する
  - 失敗時: 補助文へ即時表示したうえで、`scope=session, severity=blocker, source=settings` として既存の開始前ブロッカーへも理由を出す

- `結果クリア`
  - 対象: 中央タイムライン表示と履歴
  - 実行: 既存の `TranslatorApp._clear_result()` と同じ挙動に揃える
  - 有効条件: タイムライン表示または履歴が 1 件以上あるとき
  - 確認要否: 不要
  - 無効時の見せ方: ボタンを非活性にし、補助文へ `結果がありません` を表示する
  - 補足: 実行中の翻訳処理自体は停止しない

- `エクスポート`
  - 対象: 現在の翻訳結果テキスト
  - 実行: 既存の `TranslatorApp._export_result()` と同じく保存ダイアログ経由で `.txt` 出力する
  - 有効条件: エクスポート対象テキストが空でないとき
  - 確認要否: 保存ダイアログのみ
  - 無効時の見せ方: ボタンを非活性にし、補助文へ `出力対象が空です` を表示する
  - 失敗時: まず補助文へ即時表示し、中央状態バーにも保存不可理由を反映する。イベント分類を使う場合は親設計の `source` 列挙へ `export` を追加し、`scope=session, severity=runtime, source=export` として扱う

`詳細設定` は既に上部導線があるため、左下へ重複配置しない。
`翻訳開始` の主 CTA を弱めないため、クイック操作は補助的な見た目に留める。

## レイアウト方針

- 左カラム上部は既存どおり `開始/停止`、`聴く (PC音声)`、`話す (マイク)`、`詳細設定`、有効ストリーム、モード要約を置く
- 左カラム下部に `セッション構成` を追加する
- `セッション構成` は `方向サマリー -> 使用デバイス -> 使用バックエンド -> 構成更新時刻` の順に並べる
- その直下に `クイック操作` を配置する
- `セッション構成` ブロックは縦方向に少し伸びても破綻しにくいよう、各項目を 1 行表示にする
- `クイック操作` は左カラム幅が 360px 未満では 1 列縦並び、360px 以上では 2 列配置を基本とする
- 2 列配置時でも各ボタンの最小幅は 144px を下回らない
- `クイック操作` の補助文はボタン列の直下に固定し、中央状態バーが他状態で埋まっていても失敗が見えるようにする
- `main_controls_panel` の責務拡張として、`セッション構成` 表示と `クイック操作` の配置・活性制御もこのパネル側で担う

## データフロー補足

`セッション構成` は親設計の `main_controls_panel.apply_session_summary(...)` を拡張して更新する。
この設計で追加する最小引数は次とする。

- `direction_summary: tuple[str, str]`
- `device_summary: tuple[str, str]`
- `backend_summary: str`
- `config_updated_at: str  # "HH:MM:SS" 形式`

既存の上部コア操作・モード要約用引数とこの追加引数は、`TranslatorApp` 側で単一の `セッション構成` データにまとめてから `main_controls_panel` へ渡す。
各値の更新契機は次のとおり。

- `direction_summary`
  - 翻訳方向設定が `trace` または明示コールバックで確定したとき
- `device_summary`
  - 初期設定読み込み時
  - デバイス再読込成功時
  - 設定ウィンドウ内でデバイス選択が確定したとき
- `backend_summary`
  - STT / 翻訳バックエンド設定が `trace` または明示コールバックで確定したとき
- `config_updated_at`
  - 上記いずれかの要約値が更新されたとき

設定反映は親設計どおり `trace` コールバックを第一手段とし、設定ウィンドウ close 時の一括同期だけには依存しない。

## 不採用案

### タブ内容の一部を左下へ直接表示

- 再翻訳や議事録の内容を直接左下へ出すと、右ワークスペースとの責務が曖昧になる
- 左カラムが `セッション確認` ではなく `補助作業の入口` へ寄りすぎる

### クイック操作中心の構成

- ボタン密度が高くなり、主 CTA の視線誘導を壊しやすい
- 待機中は便利でも、実行中の価値が弱い

### 左下に実行状態や最終エラーを置く構成

- 中央状態バーと責務が競合し、状態源が二重になる
- 翻訳イベントの前後で左下要約と中央表示が食い違うリスクが高い

## 成功条件

- 左下の空きが視覚的に減る
- 待機中に `方向・デバイス・バックエンド` を左下だけで確認しやすくなる
- 実行中でも左下から現在の構成がすぐ分かり、中央状態バーとは役割がぶつからない
- `デバイス再読込`、`結果クリア`、`エクスポート` の対象と有効条件が実装前に定義されている
- `クリア` / `エクスポート` がヘッダーから削除され、左下クイック操作に統合されている
- 右ワークスペースの存在意義を損なわない

## テスト方針

- `セッション構成` の各フィールドが、設定変更後に即時反映されることを確認する
- `構成更新時刻` が `trace` または明示コールバックで更新され、設定ウィンドウ close に依存しないことを確認する
- 各クイック操作ボタンが有効条件どおり活性 / 非活性を切り替えることを確認する
- `結果クリア` と `エクスポート` が既存の `TranslatorApp._clear_result()` / `TranslatorApp._export_result()` と挙動一致することを確認する
- `デバイス再読込` 失敗時に `scope=session, severity=blocker, source=settings` 相当の UI 表示へ流れることを確認する
- ヘッダーから `クリア` / `エクスポート` が削除され、左下にのみ存在することを確認する

## 検証ゲート

- 文書整合:
  - `セッション構成` の責務が中央状態バーと競合していないこと
  - `main_controls_panel` の責務拡張と `apply_session_summary(...)` の更新契機が矛盾していないこと
- UI 挙動:
  - 最小幅付近でも `クイック操作` が主 CTA を圧迫しないこと
  - `結果クリア` と `エクスポート` の統合後も既存導線が二重化していないこと
